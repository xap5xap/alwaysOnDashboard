# Design: Vela kiosk wall presentation (day/night, exit affordance)

> Status: draft for review, 2026-06-30. Tracked by [AOD-39](https://linear.app/thexap/issue/AOD-39) (`type:design`, `area:kiosk` + `area:design-system`; milestone K-M1 "Kiosk runtime", project Kiosk Mode). It fixes the **kiosk wall-mount presentation** per [`kiosk-mode.md`](kiosk-mode.md) ([AOD-11](https://linear.app/thexap/issue/AOD-11)): the glanceable landscape wall of widget cards, the day/night dim + deep-red night visuals, the exit gesture + PIN affordance, and the pinned/locked fullscreen chrome. It follows the `type:design` deliverable convention in [`engineering-process.md`](engineering-process.md): a `design-` doc under `docs/specs/` plus rendered SVG mockups in `docs/specs/assets/`, screen-scoped tokens **specified** (not written into [`unistyles.ts`](../../apps/app/unistyles.ts)), merged via PR.
>
> **It supersedes the issue's original "Produce (Figma)" wording.** AOD-39's Produce list ("Wall layout (fresh / stale / error at scale), dusk/dawn ramp, exit hint + PIN") is honored in full; the **medium** is this repo's in-repo `design-` doc + rendered SVGs, not a Figma file, the same medium supersession [AOD-37](https://linear.app/thexap/issue/AOD-37) / [AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-20](https://linear.app/thexap/issue/AOD-20) / [AOD-17](https://linear.app/thexap/issue/AOD-17) / [AOD-21](https://linear.app/thexap/issue/AOD-21) / [AOD-27](https://linear.app/thexap/issue/AOD-27) / [AOD-28](https://linear.app/thexap/issue/AOD-28) / [AOD-29](https://linear.app/thexap/issue/AOD-29) used. No scope change, only the medium.
>
> **It is a distinct presentation profile, NOT an application of the [AOD-21](https://linear.app/thexap/issue/AOD-21) shell.** AOD-27 / AOD-28 / AOD-29 are applications of the shared nav shell ([`design-core-navigation.md`](design-core-navigation.md)); the kiosk wall is not. It is chrome-**hidden** fullscreen, the one v1 surface that lives **outside** the app bar / nav shell entirely ([`app-ia.md`](app-ia.md) §5 row 12: fullscreen, chrome hidden). [AOD-21](https://linear.app/thexap/issue/AOD-21) §9 designed the kiosk **entry** and the fullscreen entry **transition** and explicitly deferred the **wall presentation** here (§9, §13). This design is that wall: it **applies** the [AOD-11](https://linear.app/thexap/issue/AOD-11) runtime and the [AOD-21](https://linear.app/thexap/issue/AOD-21) §9 fullscreen frame as fixed inputs, and **composes** the [AOD-37](https://linear.app/thexap/issue/AOD-37) widget cards as the units it lays out. Completing it closes the entire core-screen / visual design set (shell [AOD-21](https://linear.app/thexap/issue/AOD-21) + dashboard [AOD-27](https://linear.app/thexap/issue/AOD-27) + connections [AOD-28](https://linear.app/thexap/issue/AOD-28) + onboarding/paywall [AOD-29](https://linear.app/thexap/issue/AOD-29) + this wall).
>
> **What this fixes, and what it must not touch.** It fixes **visuals only**: the wall layout as a glanceable landscape display, the day/night dim + deep-red night visuals, the exit affordance, and the pinned fullscreen chrome, each expressed as a composition of the [AOD-37](https://linear.app/thexap/issue/AOD-37) cards and the coded tokens (especially the `night.*` palette and the `overlay` dim token). It does **not** edit [`unistyles.ts`](../../apps/app/unistyles.ts), does **not** write kiosk or screen code, and does **not** scaffold the `(app)/kiosk` route ([`app-ia.md`](app-ia.md) §9 lists it as ADD; the kiosk build is a future `type:tech-task`). It does **not** re-decide the [AOD-11](https://linear.app/thexap/issue/AOD-11) runtime (keep-awake, the curve math, pinning, the exit gesture + PIN internals, the orientation lock), the IA ([AOD-17](https://linear.app/thexap/issue/AOD-17)), the tokens ([AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-66](https://linear.app/thexap/issue/AOD-66)), the brand ([AOD-18](https://linear.app/thexap/issue/AOD-18)), the shell ([AOD-21](https://linear.app/thexap/issue/AOD-21)), the components ([AOD-20](https://linear.app/thexap/issue/AOD-20)), or the card system ([AOD-37](https://linear.app/thexap/issue/AOD-37)). Unlike [AOD-28](https://linear.app/thexap/issue/AOD-28) / [AOD-29](https://linear.app/thexap/issue/AOD-29) it **does** flag one screen-scoped token group (the `wall` group, section 10), because the wall needs a type-scale multiplier the theme does not name.
>
> **Design FOR what already exists.** The ambient signal it coordinates with is **built in code** ([`AmbientContext.tsx`](../../apps/app/src/ambient/AmbientContext.tsx), [AOD-62](https://linear.app/thexap/issue/AOD-62)): the `useAmbient()` hook and the `AmbientProvider`. The [AOD-11](https://linear.app/thexap/issue/AOD-11) kiosk runtime that drives it (keep-awake, `computeAmbient`, the exit lock, pinning) has **no code yet** (the `(app)/kiosk` route is not scaffolded), so the wall is designed **ahead** of its backing, the way [AOD-29](https://linear.app/thexap/issue/AOD-29) designed onboarding ahead of its route. This design fixes how the wall looks; a future `type:tech-task` builds the runtime that renders it.

## 1. Purpose and scope

[`kiosk-mode.md`](kiosk-mode.md) ([AOD-11](https://linear.app/thexap/issue/AOD-11)) fixed the kiosk **runtime**: keep-awake, the day/night schedule + dim curve, the wall-mount profile *as a runtime concept*, the exit lock, pinning, and the Fire HD 8 worked example. [`design-core-navigation.md`](design-core-navigation.md) ([AOD-21](https://linear.app/thexap/issue/AOD-21)) §9 fixed the kiosk **entry** (the two entry points, the `Gate` to paywall handoff, the fullscreen entry transition) and deferred the **wall** here. This design lays out that wall: how the dashboard reads as a glanceable landscape display, how it dims through the day and recolours at night, how the owner exits it, and how it sits pinned and chrome-free.

It is the design counterpart of the [AOD-11](https://linear.app/thexap/issue/AOD-11) runtime spec: the spec fixed the behavior; this fixes how that behavior **looks**, expressed as compositions of the [AOD-37](https://linear.app/thexap/issue/AOD-37) cards and the coded tokens. It is the last remaining screen design; with it, the v1 screen/visual design set is complete.

It fixes exactly five things:

1. **The wall-mount profile** (section 3): the presentation contract (`theme: dark`, `typeScale`, `minContrast`, `hideChrome`) applied at render over an ordinary `DashboardLayout`, with the visual values [AOD-11](https://linear.app/thexap/issue/AOD-11) §7.1 left to the design.
2. **The wall layout** (section 4): the dashboard drawn as a glanceable landscape wall of [AOD-37](https://linear.app/thexap/issue/AOD-37) cards on the near-black field, at the wall type scale, with no app chrome.
3. **The wall at scale: fresh / stale / error** (section 5): how the [AOD-37](https://linear.app/thexap/issue/AOD-37) §5 lifecycle states read across a room, the Must-cover "fresh / stale / error at scale."
4. **Day / night on the wall** (section 6): the wall-level application of the [AOD-11](https://linear.app/thexap/issue/AOD-11) dim curve and the [AOD-37](https://linear.app/thexap/issue/AOD-37) §7 night opt-in: the whole wall dimming through the day, the deep-red night recolour, and the backlight drop. The Must-cover "dusk/dawn ramp."
5. **The exit affordance and the fullscreen chrome** (sections 7, 8): the presentation of the [AOD-11](https://linear.app/thexap/issue/AOD-11) §4.3 gesture + PIN (the Must-cover "exit hint + PIN"), and the sustained pinned/locked fullscreen frame.

Plus the cross-cutting handoffs: the ownership splits (section 9), the token addition (section 10), the reconciliation with the shipped ambient code and the [AOD-21](https://linear.app/thexap/issue/AOD-21) placeholder wall (section 11), and the seams to the build (section 12).

**In scope:** the five surfaces above, each as a visual + interaction contract expressed as a composition of the [AOD-37](https://linear.app/thexap/issue/AOD-37) cards and the coded tokens, inside the [AOD-21](https://linear.app/thexap/issue/AOD-21) §9 fullscreen frame, plus the handoffs (sections 9 to 12).

**Out of scope (named so the frame is clear):**

- **The kiosk entry chrome and the entry transition** ([AOD-21](https://linear.app/thexap/issue/AOD-21) §9, Done): the two entry points, the `canUseKiosk` `Gate` to Paywall handoff, and the push-fullscreen entry transition (`gestureEnabled: false`, OS back intercepted). This design draws the wall the entry lands on; it does not redraw the entry (section 9).
- **The kiosk runtime mechanics** ([AOD-11](https://linear.app/thexap/issue/AOD-11), Done): keep-awake (`expo-keep-awake`), the `computeAmbient` curve math and its `nightDim` anchor, the `CadenceProfile` scheduling and the night multiplier, the exit gesture + PIN internals (the `holdMs`, the hash in `expo-secure-store`, the back-intercept), pinning (`startLockTask` / Guided Access), the `expo-brightness` backlight, and the landscape orientation lock. This design **visualizes** them; it does not re-decide them (section 9).
- **The widget card system** ([AOD-37](https://linear.app/thexap/issue/AOD-37), Done): the card chrome, the six lifecycle states, the value-first body, the type ramp, the `dimsWithAmbient` overlay, the deep-red night palette, and the `useAmbient()` opt-in are **composed**, not redesigned.
- **The ambient provider** ([AOD-62](https://linear.app/thexap/issue/AOD-62), built): `useAmbient()` / `AmbientProvider` / `AMBIENT_DAY` and the `__velaSetAmbient` dev seam. The wall coordinates with the `{ phase, dimLevel }` it provides; this places the visuals, not the provider (section 9, 11).
- **The dashboard editor and the free-form layout** ([AOD-27](https://linear.app/thexap/issue/AOD-27) / [AOD-7](https://linear.app/thexap/issue/AOD-7), Done): the wall presents a layout the user built with the normal editor; the geometry (the free-form rects) is untouched (section 3).
- **The entitlement gate** ([AOD-12](https://linear.app/thexap/issue/AOD-12)): kiosk is Pro; the `canUseKiosk` `Gate` is at the entry ([AOD-21](https://linear.app/thexap/issue/AOD-21) §9), not on the wall. The wall is what a Pro user sees after the gate passes.
- **Writing kiosk or screen code, or scaffolding the `(app)/kiosk` route**: a future `type:tech-task` ([`app-ia.md`](app-ia.md) §9).

## 3. The wall-mount profile (a profile over a layout)

The wall is not a new layout. [`kiosk-mode.md`](kiosk-mode.md) §7 fixes it as a **presentation profile applied at render over an ordinary `DashboardLayout`**: the geometry (the free-form rects, [AOD-7](https://linear.app/thexap/issue/AOD-7)) is untouched, and the profile changes only the type scale, the theme, the contrast floor, and the chrome. [AOD-11](https://linear.app/thexap/issue/AOD-11) §7.1 declares the interface but leaves its **visual values** to the design; this section fixes them.

| Profile field ([AOD-11](https://linear.app/thexap/issue/AOD-11) §7.1) | Value fixed here | Rationale |
|---|---|---|
| `theme` | `"dark"` (the shipped dark theme, [AOD-19](https://linear.app/thexap/issue/AOD-19)) | The canonical ambient surface: near-black `background`, low glare in a dark room. No new palette. |
| `typeScale` | `1.4` (the `wall.typeScale` token, section 10) | Across-the-room legibility. Multiplies the [AOD-37](https://linear.app/thexap/issue/AOD-37) §3.3 type ramp uniformly, so the value-first hierarchy is preserved, just larger. |
| `minContrast` | `"AA"` | The shipped palette already meets AA; the wall enforces it as the floor so a dimmed card never drops below legible. |
| `hideChrome` | `true` | No app bar, no nav, no edit handles, no status chrome, no per-card manual refresh control. The wall shows content, not controls (section 5, 8). |

The seam this preserves (from [`kiosk-mode.md`](kiosk-mode.md) §7.2, restated so the design does not violate it): **layout geometry is untouched** (the user builds the wall layout with the normal free-form editor; the profile never moves a widget or rewrites a rect), **widget render props are untouched** (`{ data, config, size }`; the profile is host chrome, like the AOD-37 dim overlay), and **no registry edit** (turning on the wall adds no `ServiceDefinition` / `WidgetDefinition`). This design draws the profile's *look*; it does not touch the layout, host, or registry.

## 4. The wall layout

The wall is the dashboard, presented. The same `DashboardLayout` a user glances at on their phone becomes a wall display by mounting its free-form layout under the profile: near-black field, [AOD-37](https://linear.app/thexap/issue/AOD-37) cards at the wall type scale, no app chrome. The mockup uses Xavier's real dogfood wall (Clock + Linear "My issues" + Claude usage + Weather) on a Fire HD 8 in landscape.

- **The field is `colors.background`, edge to edge.** The near-black dark-theme backdrop, in landscape, with no app chrome ([AOD-21](https://linear.app/thexap/issue/AOD-21) §9 fullscreen, `hideChrome`). The wall is the brand at its purest: near-black, quiet, glanceable.
- **The cards are the [AOD-37](https://linear.app/thexap/issue/AOD-37) chrome, only larger.** Every card is the shared frame (`surface` fill, 1px `border`, `radius.md`) with its value-first body, rendered at `wall.typeScale`. The Clock is the wall hero (`type.display`, the product's origin as a Fire HD 8 wall clock). No card is redrawn; the wall scales the type and themes the field around cards the card system already produces.
- **The geometry is untouched.** The arrangement is the user's own free-form layout ([AOD-7](https://linear.app/thexap/issue/AOD-7)); the profile does not move a widget. This is the [AOD-8](https://linear.app/thexap/issue/AOD-8) seam holding for a whole feature: a wall display without a single edit to layout, dashboard, or registry internals.
- **The example is illustrative, not a fixed set.** The specific v1 widget subset is [AOD-4](https://linear.app/thexap/issue/AOD-4)'s (open); the wall presents whatever layout it is handed. The dogfood wall is the worked example ([`kiosk-mode.md`](kiosk-mode.md) §10), not a constraint.

## 5. The wall at scale: fresh / stale / error

The Must-cover names "fresh / stale / error at scale." [AOD-37](https://linear.app/thexap/issue/AOD-37) §5 owns the six lifecycle-state visuals (the dot, the captions, the prompts); this section fixes how they **read on a wall**, across a room, with no one standing at it. Three wall-specific rules follow from "glanceable, not interacted with."

![The Vela kiosk wall lifecycle states at across-the-room scale. Five widget cards drawn at wall scale: fresh draws the Claude spend value 42 dollars with a quiet header and no status dot; stale keeps the same last-known 42 dollars and adds an amber status dot with an updated six minutes ago caption; error with data keeps 42 dollars and adds a red status dot with a could not refresh caption; error with no data shows a could not load placeholder with an auto retry note; disconnected shows a connect Claude prompt with a 409 owner reconnects note. Below, three wall-scale rules: the colored status dot is the glanceable mark readable at two to three metres; the wall never blanks because stale and error keep the last-known value on screen; and prompts are informational because the wall auto-refreshes so a passer-by never taps and reconnect is the owner's action. A side note states AOD-37 owns the state vocabulary and this design fixes only the wall reading.](assets/design-kiosk-wall-states.svg)

<details>
<summary>The wall reading of the AOD-37 states &amp; tokens</summary>

```
fresh        : the AOD-37 §5 baseline; renderer draws, quiet header, no dot. The state the wall sits in almost always.
stale        : last-known value STAYS + warning dot (r 4.5, #F2B84B) + "updated Nm ago" (type.caption). The dot is the glanceable mark.
error+data   : last-known value STAYS + error dot (#FF6B6B) + "couldn't refresh". Red dot signals the refresh failed; the value holds.
error,no data: host placeholder "Couldn't load" + Retry (accent). On a wall Retry is AUTOMATIC (the wall auto-refreshes; no one taps).
disconnected : host prompt "Connect <Service>" (link glyph). Surfaces AOD-9's 409 needs_reconnect; the fix is the OWNER's reconnect.
rule 1       : the status dot (AOD-37 §3.4, r 4.5) is the primary glanceable mark at 2-3 m; the caption is the up-close detail.
rule 2       : the wall never blanks; data-bearing states keep last-known data on screen (a glanceable display shows the last good value).
rule 3       : prompts are informational, not tapped; hideChrome suppresses the manual refresh control; Retry is auto, Connect is the owner's.
```
</details>

- **The dot is the glanceable mark.** At 2 to 3 metres the [AOD-37](https://linear.app/thexap/issue/AOD-37) §3.4 status dot (`warning` amber = stale, `error` red = error) is what reads; the caption ("updated 6m ago") is the detail for when the owner steps close. On a wall the dot carries the status, exactly as the card system intended, now stated as the wall's primary signal.
- **The wall never blanks.** `stale` and `error`-with-data keep the **last-known value** on screen ([AOD-37](https://linear.app/thexap/issue/AOD-37) §5: "a glanceable display never blanks"). The wall shows the last good `$42`, dotted, not an empty card. Only `error`-no-data (the one substate with nothing to draw) shows the "Couldn't load" placeholder.
- **Prompts are informational, not tapped.** Because `hideChrome` suppresses the per-card manual refresh control (there is no one to tap it) and the wall auto-refreshes on the kiosk cadence ([`kiosk-mode.md`](kiosk-mode.md) §6), the `Retry` action is effectively **automatic** and the `Connect` prompt is a **status**, not a call to action for a passer-by. `disconnected` surfaces [AOD-9](https://linear.app/thexap/issue/AOD-9)'s `409 needs_reconnect` unchanged; the fix is the owner reconnecting the service, not a tap on the wall. The prompt vocabulary is [AOD-37](https://linear.app/thexap/issue/AOD-37)'s; the wall reads it as information.

## 6. Day / night on the wall (the dusk to dawn ramp)

The Must-cover names the "dusk/dawn ramp." Two upstream designs own the pieces: [`kiosk-mode.md`](kiosk-mode.md) §8 owns the **schedule and dim curve** (`computeAmbient(now) → { phase, dimLevel }`, the `nightDim ~0.7` anchor), and [`design-widget-system.md`](design-widget-system.md) §7 owns the **per-widget** reactions (the global overlay §7.1, the deep-red `useAmbient()` opt-in §7.3, the backlight named §7.4). This section fixes the one thing neither did: the **wall-level application** of both, the whole wall dimming through the day. It consumes the curve and the reactions; it does not rebuild them.

![The Vela kiosk wall at day, dusk, and night. Three landscape mini-walls show the same wall. At day, dimLevel zero, every card and the clock render light on the near-black field. At dusk, dimLevel about 0.35, a global overlay uniformly darkens the default cards. At night, dimLevel about 0.70, the overlay darkens the default cards further, the clock opts out and recolours itself deep red, and a badge notes the device backlight drops to about 0.3. A slim curve strip beneath ties each wall to its point on the AOD-11 dim curve, easing dimLevel from zero in the day up to about 0.7 at night with dayStart at 07:00 and nightStart at 21:00. A three-layer legend names the global overlay that darkens every dimsWithAmbient card with no code, the deep-red useAmbient opt-in that recolours the clock because the overlay can only darken, and the kiosk-only backlight reduction. A footer states AOD-11 owns the curve and schedule, AOD-37 owns the per-widget overlay and deep-red palette, and this shows only the wall-level application.](assets/design-kiosk-daynight.svg)

<details>
<summary>The three layers &amp; the dim tokens</summary>

```
day (dimLevel 0)   : full brightness. Every card and the Clock render in the standard palette on colors.background.
dusk (dimLevel~0.35): the global overlay eases in; host paints overlay.color (#000) at dimLevel × overlay.maxDim (0.72) over each dimsWithAmbient card.
night (dimLevel~0.70): overlay near full on default cards + the Clock recolours to night.* (opt-in) + the backlight drops. The three layers compose.
layer 1 (overlay)  : AOD-37 §7.1; every dimsWithAmbient:true card darkens uniformly, no widget code. overlay.maxDim 0.72 anchors "full night".
layer 2 (deep-red) : AOD-37 §7.3; the Clock sets dimsWithAmbient:false, reads useAmbient(), draws night.primary/secondary/muted on night.surface.
layer 3 (backlight): AOD-11 §8.3; expo-brightness drops the real backlight (kiosk-only), a light reduction the overlay cannot do. Named, not owned here.
curve              : AOD-11 §8.4 computeAmbient(now) eases dimLevel 0 → ~0.7 with a smoothstep ramp at dayStart / nightStart. CONSUMED, not built.
```
</details>

- **Layer 1, the global overlay (the whole wall).** Every `dimsWithAmbient: true` card ([AOD-37](https://linear.app/thexap/issue/AOD-37) §7.1) darkens uniformly as `dimLevel` rises: the host paints `overlay.color` at `dimLevel × overlay.maxDim`. On the wall this is the whole field of cards receding together at dusk, no per-widget code. `overlay.maxDim` (`0.72`) matches the curve's `nightDim ~0.7` so full night reads as deep dim, not black.
- **Layer 2, the deep-red opt-in (the Clock).** The Clock ([AOD-37](https://linear.app/thexap/issue/AOD-37) §7.3, §8.5) sets `dimsWithAmbient: false`, reads `useAmbient()`, and at `phase === 'night'` recolours to the `night.*` palette (deep red on red-black), because the overlay can only darken, not recolour. On the wall this is the one iconic night element: a glowing-red wall clock over a field of dimmed cards. It dims further as `dimLevel` rises.
- **Layer 3, the backlight (kiosk-only, named).** [AOD-11](https://linear.app/thexap/issue/AOD-11) §8.3 drives `expo-brightness` down at night, a real light reduction the overlay cannot achieve. This is a **runtime** capability, not a widget visual; the wall composes with it (darker pixels **and** deeper red **and** less backlight) but does not own it. The design shows it as the third layer so the full night look is honest.
- **The curve is consumed, not built.** `computeAmbient(now)` ([AOD-11](https://linear.app/thexap/issue/AOD-11) §8.4) produces `{ phase, dimLevel }`; the wall is what those values look like at day, dusk, and night. This design does not draw the schedule UI or the curve math (those are [AOD-11](https://linear.app/thexap/issue/AOD-11)'s and are already diagrammed in [`kiosk-daynight-ambient.svg`](assets/kiosk-daynight-ambient.svg)); it draws the wall-level result. This complements [`design-ambient-dim.svg`](assets/design-ambient-dim.svg) ([AOD-37](https://linear.app/thexap/issue/AOD-37)), which shows a **single card's** two reactions; the wall mockup shows the **whole wall** plus the backlight.

## 7. The exit affordance

The Must-cover names "exit hint + PIN." [`kiosk-mode.md`](kiosk-mode.md) §4.3 owns the exit **mechanic**: the `long_press_corner` gesture, the `holdMs` deliberate hold, the PIN verified against a hash in `expo-secure-store`, and the back-intercept. This section fixes the **affordance**: how the exit is presented on a wall that must not be dismissed by a stray tap yet must let the owner leave.

![The Vela kiosk exit affordance in three landscape wall frames. First, the wall carries an invisible long-press exit corner in the bottom right, a faint dashed hit-target with no visible control, so a passer-by sees only content. Second, a deliberate long-press for about two seconds fills a blue progress ring at that corner and reveals a Hold to exit hint pill. Third, completing the hold raises a PIN surface: the wall dims under a scrim and a centered card reads Enter PIN with four dots and a numeric keypad, verified against a secure-store hash. An outcomes strip states a wrong PIN shakes the card and the wall stays active, and a correct PIN releases pinning, restores brightness, unmounts the profile, and returns to the Dashboard. A footer states AOD-11 owns the exit mechanic and this design fixes only the affordance, that the OS back is intercepted so the gesture and PIN is the only exit, and that the corner is invisible by default so the wall advertises nothing at rest.](assets/design-kiosk-exit.svg)

<details>
<summary>The exit flow &amp; tokens</summary>

```
1 invisible corner : a long-press hit-target (wall.exitCorner, ~spacing(14)=56) in the trailing-bottom corner. NO visible control at rest; a wall must not advertise its exit to a passer-by.
2 hold             : on touch-down, reveal a "Hold to exit" hint pill (surface + border) and fill an accent progress ring over holdMs (AOD-11, e.g. 2000ms). A deliberate hold, not a brush.
3 PIN              : the wall dims under `scrim` (the AOD-20 §9 token) + a centered PIN card (surface + border + radius); "Enter PIN", 4 dots (accent filled), a numeric keypad (wall.pinKey keys). Verified vs the expo-secure-store hash (AOD-11 §4.3).
wrong PIN          : the card shakes (error feedback), the wall STAYS active. A wrong entry never leaves the wall.
correct PIN        : release pinning, restore system brightness, unmount the profile, return to the Dashboard (the replace back to the hub, app-ia §6).
design decision    : reveal the hint ONLY on touch-down at the corner; at rest the wall shows content, nothing else (looked at, not interacted with).
```
</details>

- **The corner is invisible by default.** The exit begins with a `long_press_corner` gesture ([`kiosk-mode.md`](kiosk-mode.md) §4.3) in the trailing-bottom corner, sized `wall.exitCorner` (section 10). The design decision the wall adds: **the corner shows no control at rest.** A wall display advertises nothing; a passer-by sees only content and never finds the exit. This is the visual counterpart of "not dismissed by a stray tap."
- **The hold reveals the hint.** On touch-down at the corner, a "Hold to exit" hint pill appears and an `accent` progress ring fills over `holdMs`. The reveal-on-touch is what keeps the affordance discoverable by the owner (who knows to press-and-hold the corner) but invisible to everyone else. The `holdMs` is [AOD-11](https://linear.app/thexap/issue/AOD-11)'s; the ring is the affordance.
- **The PIN surface is a scrim + a card.** Completing the hold dims the wall under the `scrim` token and raises a centered PIN card: "Enter PIN", four dots, a numeric keypad (`wall.pinKey` keys). This reuses the same floating-surface vocabulary as the [AOD-20](https://linear.app/thexap/issue/AOD-20) §9 / [AOD-21](https://linear.app/thexap/issue/AOD-21) §7 modal (scrim over content, a surface above), scaled for a wall touch target. The PIN is verified against the `expo-secure-store` hash ([AOD-11](https://linear.app/thexap/issue/AOD-11) §4.3); the plaintext is never stored.
- **The outcomes are the mechanic's, shown.** A wrong PIN shakes the card and the wall stays active; a correct PIN reverses enter (release pinning, restore brightness, unmount the profile, `CadenceProfile = "foreground"`, `deactivateKeepAwake`) and replaces back to the Dashboard ([`app-ia.md`](app-ia.md) §6.1). Because the OS back is intercepted ([AOD-11](https://linear.app/thexap/issue/AOD-11) §4.3, `gestureEnabled: false`), this gesture + PIN is the **only** exit. The design fixes the affordance; the reversal is [AOD-11](https://linear.app/thexap/issue/AOD-11)'s.

## 8. The fullscreen / pinned chrome

The wall lives fullscreen. [AOD-21](https://linear.app/thexap/issue/AOD-21) §9 fixed the **entry** transition (push fullscreen, chrome hidden, `gestureEnabled: false`); this section fixes the **sustained** presentation: how the wall sits pinned once entered, edge to edge, with an invisible lock, and the honest per-platform account of what pinning can enforce ([`kiosk-mode.md`](kiosk-mode.md) §9).

![The Vela kiosk sustained fullscreen chrome. On the left, the wall lives edge to edge: a landscape frame with no OS status bar, no home indicator, and no app header or nav; the background paints to the very edge while content sits inside the safe-area insets, and the only affordance is the invisible exit corner. On the right, a panel states pinning is best-effort, not a lock, with three rows: Android and Fire user screen pinning blocks a casual tap but the owner can leave, marked best-effort; Android device-owner lock task is a true lock but needs provisioning; iOS Guided Access is user-initiated with no App Store API to force it. A conclusion states the app-level gesture and PIN exit is the portable guard because it is enforced in the app not the OS, and pinning is a convenience layered on top. A footer states AOD-11 section 9 owns the pinning mechanics, AOD-21 section 9 owns the entry transition, and this fixes only the sustained fullscreen presentation.](assets/design-kiosk-chrome.svg)

<details>
<summary>The fullscreen frame, pinning honesty &amp; tokens</summary>

```
frame     : colors.background edge to edge · NO OS status bar / home indicator · NO app header / nav (hideChrome). Background paints to the edge; content sits in the safe-area insets.
lock       : INVISIBLE; the wall shows no lock chrome and no controls. The only affordance is the invisible exit corner (section 7). A wall shows content, not controls.
pinning    : best-effort, NOT a lock (AOD-11 §9). Android/Fire user screen pinning (best-effort · owner exits via Back+Overview or the app gesture+PIN); Android device-owner lock task (true lock · provisioned only); iOS Guided Access (user-initiated · no App Store API).
guard      : the app-level gesture + PIN exit (section 7) is the PORTABLE guard; it holds on every platform because it is enforced in the app, not the OS. Pinning is a convenience layered on top, never the lock.
split      : the ENTRY transition (push fullscreen, chrome hidden, gestureEnabled:false) is AOD-21 §9; the SUSTAINED wall is this design. Not redrawn.
```
</details>

- **Edge to edge, no OS chrome.** The wall fills the display: no status bar, no home indicator, no app header or nav (`hideChrome` + the [AOD-21](https://linear.app/thexap/issue/AOD-21) §9 fullscreen frame). The `background` paints to the very edge; content sits inside the safe-area insets so nothing is clipped by a notch or a rounded corner. This is the "always-on ambient display" the product promises: content, uninterrupted.
- **The lock is invisible.** There is no visible lock chrome, no controls, no "you are in kiosk mode" banner. The wall shows content and nothing else; the only affordance is the invisible exit corner (section 7). A wall that advertised its lock would invite tampering; one that shows only content is glanceable and calm.
- **Pinning is honest, per platform.** [`kiosk-mode.md`](kiosk-mode.md) §9 is emphatic that no consumer install can force itself pinned on an unsupervised device. The design surfaces this honestly: Android/Fire user screen pinning is best-effort (a casual tap is blocked; the owner can still leave), device-owner lock task is a true lock but needs provisioning, and iOS Guided Access is user-initiated with no app API. These are informational states the setup flow explains, not something the wall enforces.
- **The app gesture + PIN is the portable guard.** The load-bearing conclusion: the app-level exit (section 7) is what actually stops a passer-by, because it is enforced **in the app**, not by the OS, and works identically on every platform. Pinning is a convenience layered on top, never the lock. This design draws the guard (section 7) and states the honesty (here); the mechanics are [AOD-11](https://linear.app/thexap/issue/AOD-11)'s.

## 10. Screen-scoped token addition (the `wall` group)

The wall composes almost entirely from the coded tokens (`background`, `surface`, `border`, `text`, `textMuted`, `accent`, the `night.*` palette, the `overlay` dim token, `scrim`, the status colors, `spacing`, `radius`, the `type` ramp). Where it needs geometry the theme does not name, it adds one **screen-scoped component group**, the way [AOD-20](https://linear.app/thexap/issue/AOD-20) added `button` / `input`, [AOD-37](https://linear.app/thexap/issue/AOD-37) added `clockSize` / `weatherIcon`, and [AOD-21](https://linear.app/thexap/issue/AOD-21) flagged `appBar` / `screen`. These are **specified here, not coded**; the build adds them to [`unistyles.ts`](../../apps/app/unistyles.ts).

| # | Addition | Kind | What it carries |
|---|---|---|---|
| 1 | `wall` group | **Additive (component).** | `typeScale` (`1.4`, the [AOD-11](https://linear.app/thexap/issue/AOD-11) §7.1 `WallMountProfile.typeScale` multiplier on the `type.*` ramp) · `exitCorner` (`spacing(14)` = 56, the invisible long-press exit hit-target) · `pinKey` (`spacing(16)` = 64, the PIN-pad key diameter on the exit surface). Numbers only. |
| 2 | The night wall | **Composes, no new token.** | The `night.*` palette + the `overlay` dim token ([AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-37](https://linear.app/thexap/issue/AOD-37) §3.2 / §3.4), already coded. The wall applies them; it adds none. |
| 3 | The PIN surface | **Composes, no new token.** | `scrim` + a `surface` / `border` / `radius` card ([AOD-20](https://linear.app/thexap/issue/AOD-20) §9), the same floating-surface vocabulary as the modal. No new token. |
| 4 | The fullscreen frame | **Composes, no new token.** | `background` edge to edge + `rt.insets` safe-area (a runtime value, not a token) + `hideChrome` (a profile field, not a token). No new token. |

The `wall` group carries only **numbers + a multiplier on `type.*`** (it does not embed a `TextStyle`), so it is Unistyles-safe, the same discipline the `aod-unistyles-style-token-gotcha` build note recorded for `appBar` / `screen` / `elevation` and the [AOD-20](https://linear.app/thexap/issue/AOD-20) groups: plain numbers and role-name references, no deep `type.*` object in the theme. `typeScale` is the load-bearing value (it is the one field that changes the render, per section 3); `exitCorner` and `pinKey` size the exit surface (section 7). The build multiplies the [AOD-37](https://linear.app/thexap/issue/AOD-37) §3.3 rendered font sizes by `wall.typeScale` at the wall profile; it does not fork the type ramp.
