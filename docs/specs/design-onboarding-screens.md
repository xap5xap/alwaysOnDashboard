# Design: Vela onboarding screens + the paywall body

> Status: draft for review, 2026-06-30. Tracked by [AOD-29](https://linear.app/thexap/issue/AOD-29) (`type:design`, `area:onboarding`; milestone PS-M3 "First vertical slice (Linear My Issues)", project Platform & App Shell). The **third and final per-feature core-screen design**: the first-run onboarding sequence, the Sign In interior, the empty / success states, and the full paywall body, laid out inside the [AOD-21](https://linear.app/thexap/issue/AOD-21) shell. It follows the `type:design` deliverable convention in [`engineering-process.md`](../engineering-process.md): a `design-` doc under `docs/specs/` plus rendered SVG mockups in `docs/specs/assets/`, screen-scoped tokens **specified** (not written into [`unistyles.ts`](../../apps/app/unistyles.ts)), merged via PR.
>
> **It supersedes the issue's original "Produce (Figma) ... using DS-4 components" wording, and resolves its planning-code references.** AOD-29 was written from the [`batch-4-gate-issues.md`](../planning/batch-4-gate-issues.md) roadmap, before [AOD-37](https://linear.app/thexap/issue/AOD-37) set the repo's `type:design` convention (a `design-` doc + rendered SVGs in-repo, not a Figma file). This deliverable follows that established convention, the same medium supersession [AOD-37](https://linear.app/thexap/issue/AOD-37) / [AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-20](https://linear.app/thexap/issue/AOD-20) / [AOD-17](https://linear.app/thexap/issue/AOD-17) / [AOD-21](https://linear.app/thexap/issue/AOD-21) / [AOD-27](https://linear.app/thexap/issue/AOD-27) / [AOD-28](https://linear.app/thexap/issue/AOD-28) used. The issue's planning codes resolve as: **"DS-4 components" are the [AOD-20](https://linear.app/thexap/issue/AOD-20) component library** ([`design-component-library.md`](design-component-library.md)); **"the chosen model (PS-3)" is [AOD-24](https://linear.app/thexap/issue/AOD-24)** (`[Decision] Onboarding model: guided vs free-form`) and **"the flow spec (PS-4)" is [AOD-26](https://linear.app/thexap/issue/AOD-26)** (`[Spec] Onboarding + connect-service flow`), both still in Backlog. Their substance is **already fixed by the IA** ([`app-ia.md`](app-ia.md) §3 the auth-gated zones, §4.2 the gate / first-run model, §7 the connect-service entry), so this design applies the IA as the locked model / flow source and treats AOD-24 / AOD-26 as open-but-subsumed upstream (section 8). No scope change, only the medium and the reference resolution.
>
> **It is an application of the [AOD-21](https://linear.app/thexap/issue/AOD-21) shell, not a redesign of it.** [`design-core-navigation.md`](design-core-navigation.md) ([AOD-21](https://linear.app/thexap/issue/AOD-21)) designed the shared navigation **shell** once and explicitly handed three interiors here: §4 / §12 placed the **Sign In auth card** and deferred its **interior** (the form, the sign-up affordance, the errors); §4 drew the **onboarding zone** and deferred its **interior**; §7 fixed the **paywall modal presentation** (scrim + dialog at `elevation.overlay`) and §13 handed the **full paywall body** here. This design fills those three. It is the exact relationship [AOD-27](https://linear.app/thexap/issue/AOD-27) (the dashboard interior) and [AOD-28](https://linear.app/thexap/issue/AOD-28) (the connections interior) have to the same shell, and the relationship the per-widget designs have to [`design-widget-system.md`](design-widget-system.md) ([AOD-37](https://linear.app/thexap/issue/AOD-37)).
>
> **What this fixes, and what it must not touch.** It fixes **visuals only**: the Sign In form interior, the first-run onboarding sequence, the empty / success states, and the paywall body, each expressed as a composition of the [AOD-20](https://linear.app/thexap/issue/AOD-20) components and the coded tokens, laid out inside the [AOD-21](https://linear.app/thexap/issue/AOD-21) shell. It does **not** edit [`unistyles.ts`](../../apps/app/unistyles.ts), does **not** write screen code, and does **not** scaffold the onboarding / paywall routes ([`app-ia.md`](app-ia.md) §9 lists them as ADD; a future `type:tech-task`). It does **not** re-open the IA ([AOD-17](https://linear.app/thexap/issue/AOD-17)), the tokens ([AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-66](https://linear.app/thexap/issue/AOD-66)), the brand ([AOD-18](https://linear.app/thexap/issue/AOD-18)), the shell ([AOD-21](https://linear.app/thexap/issue/AOD-21)), or the components ([AOD-20](https://linear.app/thexap/issue/AOD-20)). Like [AOD-28](https://linear.app/thexap/issue/AOD-28), it composes entirely from existing tokens and component groups and **adds no new token group** (section 9).
>
> **It states the paywall overlap, it does not silently claim it.** AOD-29's one-line Produce list is onboarding-centric and does not name the paywall, yet every Done sibling routes the full paywall **body** here ([AOD-20](https://linear.app/thexap/issue/AOD-20) §11 / §14, [AOD-21](https://linear.app/thexap/issue/AOD-21) §13, [AOD-27](https://linear.app/thexap/issue/AOD-27) §12, [AOD-28](https://linear.app/thexap/issue/AOD-28) §8 / §11). A separate planning issue, [AOD-38](https://linear.app/thexap/issue/AOD-38) (`[Design] Paywall, upsell, lock overlays, downgrade dialog`, Billing project, Backlog), also nominally owns it. This design **includes the paywall body** (the consistent home the whole design corpus assigns it, and the surface the onboarding trial CTA leads into) and resolves the split explicitly in section 8: [AOD-21](https://linear.app/thexap/issue/AOD-21) owns the presentation, [AOD-12](https://linear.app/thexap/issue/AOD-12) owns the math / triggers, [AOD-20](https://linear.app/thexap/issue/AOD-20) owns the lock affordance, **AOD-29 owns the body**, and [AOD-38](https://linear.app/thexap/issue/AOD-38) reduces to the lapse **downgrade / freeze dialog** that onboarding never reaches.
>
> **Design FOR what already exists.** The Sign In card is **built in code** ([`SignIn.tsx`](../../apps/app/src/screens/SignIn.tsx)); this design **canonicalizes** its interior and flags every extension **additively** (section 10), not a greenfield redraw. Onboarding and the paywall have **no code yet** (the routes are `_layout` / `index` / `settings` only, [`app-ia.md`](app-ia.md) §9), so they are designed **ahead** of their backing, the way [AOD-27](https://linear.app/thexap/issue/AOD-27) designed the dashboards switcher ahead of its multi-dashboard table; the entitlement surface they coordinate with **is** built ([`Gate.tsx`](../../apps/app/src/entitlements/Gate.tsx), [`CustomerInfoContext.tsx`](../../apps/app/src/entitlements/CustomerInfoContext.tsx), [`useEntitlements.ts`](../../apps/app/src/entitlements/useEntitlements.ts)).

## 1. Purpose and scope

[`app-ia.md`](app-ia.md) (the IA) fixed where Sign In, Onboarding, and the Paywall live and how each is reached (surfaces 1, 2, 13). [`design-core-navigation.md`](design-core-navigation.md) (the shell) fixed the `(auth)` zone, the onboarding zone, the gate, and the paywall modal presentation around them, and deferred their **interiors** here. This design lays out those interiors: how a user signs in, how a brand-new user is walked through first-run setup, how the empty and success moments read, and how the upgrade paywall is laid out.

It is to the auth / onboarding zones what [AOD-27](https://linear.app/thexap/issue/AOD-27) is to the Dashboard and [AOD-28](https://linear.app/thexap/issue/AOD-28) is to Connections: an **application** of a fixed shell, composing the [AOD-20](https://linear.app/thexap/issue/AOD-20) components, not an invention of new chrome. Completing it closes the core-screen design set (shell + dashboard + connections + onboarding / paywall).

It fixes exactly five things:

1. **The Sign In interior** (section 3): the email / password form, the primary sign-in, the "Create an account" sign-up affordance, and the busy / error states, inside the [AOD-20](https://linear.app/thexap/issue/AOD-20) §8 auth card the [AOD-21](https://linear.app/thexap/issue/AOD-21) §12 shell placed.
2. **The onboarding first-run sequence** (section 4): the welcome, the connect-your-first-service step (composing the [AOD-28](https://linear.app/thexap/issue/AOD-28) connections surface), and the finish into the Dashboard, with the stepper and the skip path.
3. **The onboarding empty / success states** (section 5): the calm "nothing connected yet" empty, the "service connected" success, and the finish handoff into the [AOD-27](https://linear.app/thexap/issue/AOD-27) §5 empty dashboard. The explicit Must-cover.
4. **The paywall body** (section 6): the 7-day-trial lead, the Pro value props, the monthly / annual packages, the Upgrade CTA, and restore purchases, inside the [AOD-21](https://linear.app/thexap/issue/AOD-21) §7 modal presentation.
5. **The paywall per-trigger framing** (section 7): how the one paywall route leads with the matching upsell angle by its `trigger` param (`kiosk` / `themes` / `services` / `dashboards` / `refresh`, [AOD-12](https://linear.app/thexap/issue/AOD-12) §9).

Plus the cross-cutting handoffs: the ownership splits and the planning-code resolutions (section 8), the token note (section 9), the reconciliation with the shipped Sign In (section 10), and the seams to the build and the sibling issues (section 11).

**In scope:** the five surfaces above, each as a visual + interaction contract expressed as a composition of the [AOD-20](https://linear.app/thexap/issue/AOD-20) components and the coded tokens, laid out inside the [AOD-21](https://linear.app/thexap/issue/AOD-21) shell, plus the handoffs (sections 8 to 11).

**Out of scope (named so the frame is clear):**

- **The nav shell / global chrome, the `(auth)` zone chrome, the gate logic, and the screen-level state visuals** ([AOD-21](https://linear.app/thexap/issue/AOD-21) + [AOD-17](https://linear.app/thexap/issue/AOD-17), Done): the auth card chrome, the onboarding zone frame, the gate's `(session, onboarded)` routing, the splash / loading / error visuals, and the paywall modal **presentation**. This design fills the interiors; it does not redraw the frames (section 8).
- **The component library** ([AOD-20](https://linear.app/thexap/issue/AOD-20), Done): the auth card, buttons, inputs, sheets, badges, and the lock / PRO overlay are **composed**, not redesigned.
- **The connections interior, including the credential sheet** ([AOD-28](https://linear.app/thexap/issue/AOD-28), Done): the connect-first-service step **composes** the connection rows, the connect paths, and the credential sheet; it does not redraw them (section 4).
- **The dashboard interior and the empty-dashboard CTA** ([AOD-27](https://linear.app/thexap/issue/AOD-27), Done): onboarding **finishes into** the empty dashboard; it does not redraw it (section 5).
- **The widget card system** ([AOD-37](https://linear.app/thexap/issue/AOD-37), Done).
- **The entitlement math, the tier boundaries, and the trigger / `Gate` semantics** ([AOD-12](https://linear.app/thexap/issue/AOD-12), Done) beyond placing the paywall body and the trial framing; the **RevenueCat purchase wiring** (the build); the **lapse downgrade / freeze dialog** ([AOD-38](https://linear.app/thexap/issue/AOD-38) / the billing build, section 8).
- **The OAuth broker internals** ([AOD-9](https://linear.app/thexap/issue/AOD-9)) beyond placing the connect handoff [AOD-28](https://linear.app/thexap/issue/AOD-28) already designed; the **kiosk wall** ([AOD-39](https://linear.app/thexap/issue/AOD-39) / [AOD-11](https://linear.app/thexap/issue/AOD-11)).
- **Re-deciding** the IA ([AOD-17](https://linear.app/thexap/issue/AOD-17)), the tokens ([AOD-19](https://linear.app/thexap/issue/AOD-19) / [AOD-66](https://linear.app/thexap/issue/AOD-66)), the brand ([AOD-18](https://linear.app/thexap/issue/AOD-18)), the shell ([AOD-21](https://linear.app/thexap/issue/AOD-21)), or the components ([AOD-20](https://linear.app/thexap/issue/AOD-20)).

## 3. The Sign In interior

The `(auth)` zone has one surface, Sign In ([`app-ia.md`](app-ia.md) §5 row 1). [AOD-21](https://linear.app/thexap/issue/AOD-21) §4 / §12 placed the [AOD-20](https://linear.app/thexap/issue/AOD-20) §8 auth card in that zone and deferred its **interior** here. This section fills the form: the fields, the primary action, the sign-up affordance, and the busy / error states. It canonicalizes the shipped [`SignIn.tsx`](../../apps/app/src/screens/SignIn.tsx) (section 10), which already does all of this functionally.

The first is the default state: the auth card with the vela star and wordmark, a Sign in to your dashboard subtitle, an email field, a password field, a Sign in primary button, and a Create an account secondary action, tagged card equals AOD-20 §8 and zone equals AOD-21. The second is the submitting and error state, with an Invalid login credentials error line above the primary and the primary showing a spinner reading Signing in. The third is the sign-up mode, with a Create your account subtitle, the same fields, a password hint, a Create account primary, and a Sign in instead secondary. Below, a token block names the field, primary, toggle, and error roles and states that AOD-21 owns the card and the zone while this fills the form interior only.](assets/design-onboarding-signin.svg)

<details>
<summary>The form, the modes &amp; tokens</summary>

```
card     : AOD-20 §8 authCard (surface + border + radius.lg) + the vela wordmark (brand §4). Placed by AOD-21 §12; NOT redrawn.
fields   : two AOD-20 §6 inputs (surfaceAlt fill · border · placeholder → textMuted) · email (keyboardType, no autocap) + password (secureTextEntry).
primary  : AOD-20 §5 primary (accent fill · onAccent label). Sign in (sign-in mode) / Create account (sign-up mode).
toggle   : AOD-20 §5 ghost/secondary text action that flips the mode: "Create an account" ⇄ "Sign in instead". One card, two modes.
busy     : the primary shows a spinner (onAccent) and the control is non-interactive (the shipped ActivityIndicator), the AOD-21 §8 in-button load.
error    : the auth error string (type.meta / colors.error) above the primary; clears on the next submit. The shipped result.error.
config   : a Supabase-not-configured warning (type.meta / colors.warning) stays as a dev affordance (SignIn.tsx), shown only when unconfigured.

- **The card is the [AOD-20](https://linear.app/thexap/issue/AOD-20) §8 auth card, placed by [AOD-21](https://linear.app/thexap/issue/AOD-21).** A `surface` + `border` panel at `radius.lg` with the lowercase **vela** wordmark ([`design-brand.md`](design-brand.md) §4), centered in the `(auth)` zone. This design draws only the **interior**; the card chrome and the zone are the shell's.
- **Two fields, one primary, one mode toggle.** The email and password fields are the [AOD-20](https://linear.app/thexap/issue/AOD-20) §6 input (one `surfaceAlt` fill, `placeholder -> textMuted`); the primary is the §5 primary (`accent` fill, `onAccent` label). The shipped "Create an account" action becomes the **mode toggle**: tapping it flips the card from sign-in to sign-up (the subtitle, the primary label, and the secondary swap; the fields are unchanged), so one card serves both `signInWithPassword` and `signUpWithPassword` without a second screen.
- **Busy is the in-button spinner.** While `signInWithPassword` / `signUpWithPassword` awaits, the primary shows a spinner (`onAccent`) and goes non-interactive, the [AOD-21](https://linear.app/thexap/issue/AOD-21) §8 loading treatment at button scale. This is exactly the shipped `busy` `ActivityIndicator`, themed.
- **Error is a quiet line, not an alert.** The auth error (`result.error`) renders as a `type.meta` / `colors.error` line above the primary and clears on the next submit. No modal, no toast; the brand's calm voice keeps a failed login low-key.
- **The sign-up affordance is additive, not a new screen.** The IA has one `(auth)` surface; sign-up is a **mode** of it, not a route. This canonicalizes the shipped inline "Create an account" rather than adding a Sign Up screen, keeping the `(auth)` zone single-surface (section 10).

## 4. The onboarding first-run sequence

A brand-new signed-in user who has not completed first-run setup is routed by the gate to Onboarding ([`app-ia.md`](app-ia.md) §4.2, "session, not onboarded"). This section lays out that sequence: a welcome, a connect-your-first-service step that **composes** the [AOD-28](https://linear.app/thexap/issue/AOD-28) connections surface, and a finish that replaces into the Dashboard. It is a guided-but-skippable sequence (section 8 explains why "skippable" matters for the open [AOD-24](https://linear.app/thexap/issue/AOD-24) model decision).

![The Vela onboarding first-run sequence across three device frames with a three-dot stepper along the top. The first frame is the welcome step: the vela star and wordmark centered, an Always on, never loud tagline, a Get started primary, and a Skip for now ghost. The second frame is connect your first service, composing the AOD-28 connections list: a Connect your first service heading, rows for Linear and Google Calendar offering Connect, Claude usage offering Add key, and Weather offering Set location, and a Skip for now ghost, tagged composes AOD-28. The third frame is you are set, with a success check, a Linear is connected line, and a Go to dashboard primary, tagged finishes into the AOD-27 empty dashboard. Below, a flow line runs welcome to connect, which is optional, to finish to the AOD-27 dashboard, with a dashed skip arc bypassing connect straight to the empty dashboard, and a token block notes the sequence composes AOD-28 and adds no new token.](assets/design-onboarding-sequence.svg)

<details>
<summary>The three steps, the stepper &amp; the skip path</summary>

```
presentation : (app)/onboarding, a pushed screen (app-ia §4.4 / §5 row 2). Entered by the gate; left by REPLACE into the Dashboard (app-ia §6.1).
stepper      : 3 dots (accent = current · border = pending), top of each step. No new token (the AOD-37 §3.1 dot vocabulary at chrome scale).

2 connect    : "Connect your first service." (calm voice) · the AOD-28 connection rows (one per connectableServices()) with the §4 affordance
               (Connect / Add key / Set location, by authClass) · a ghost "Skip for now". COMPOSES AOD-28; the rows + connect paths are NOT redrawn.
3 finish     : the §5 success state (a §5.1 check + "You are set." + the connected service) · a primary "Go to dashboard".
skip         : "Skip for now" on welcome or connect replaces straight into the Dashboard (the AOD-27 §5 empty state). Onboarding is never a wall.

- **Three steps, a stepper, a replace.** Onboarding is the `(app)/onboarding` pushed screen ([`app-ia.md`](app-ia.md) §5 row 2). A three-dot stepper marks progress; finishing **replaces** into the Dashboard ([`app-ia.md`](app-ia.md) §6.1), so the OS back gesture never returns into a completed onboarding. The welcome sets the **vela** wordmark and the brand tagline ([`design-brand.md`](design-brand.md) §7.2, "Always on. Never loud."), the one place onboarding leans on the brand.
- **The connect step composes [AOD-28](https://linear.app/thexap/issue/AOD-28), it does not redraw it.** The "Connect your first service" step renders the [AOD-28](https://linear.app/thexap/issue/AOD-28) connection rows (one per [`registry.connectableServices()`](../../apps/app/src/registry/registry.ts)) with the [AOD-28](https://linear.app/thexap/issue/AOD-28) §4 affordance derived from `authClass`: `Connect` for `oauth2` (Linear, Google Calendar), `Add key` for `admin_key` (Claude usage), `Set location` for `platform_key` (Weather). Tapping a row runs the [AOD-28](https://linear.app/thexap/issue/AOD-28) §5 connect path (the system-browser OAuth round-trip to `vela://oauth/done`, or the §6 credential sheet). This design **places the first-connect moment**; the rows, the connect paths, and the credential sheet are [AOD-28](https://linear.app/thexap/issue/AOD-28)'s and are reused, not redrawn.
- **The finish is the success state into the Dashboard.** Connecting a first service shows the section 5 success ("You are set." + the connected service) and a primary "Go to dashboard" that replaces into the Dashboard, where the [AOD-27](https://linear.app/thexap/issue/AOD-27) §5 empty state then invites the first widget.
- **It is skippable, by design.** "Skip for now" on the welcome or the connect step replaces straight into the Dashboard (the [AOD-27](https://linear.app/thexap/issue/AOD-27) §5 empty state), so onboarding is never a wall and a returning intent is never trapped. This honors both a guided stance (the default path connects a service) and a free-form stance (skip lands on the empty dashboard) without hard-deciding the open [AOD-24](https://linear.app/thexap/issue/AOD-24) model question (section 8).
- **The "onboarded" flag is a build seam.** Finishing or skipping sets the [`app-ia.md`](app-ia.md) §4.2 "onboarded" signal so the gate routes to the Dashboard on the next launch. Where that flag lives ( `user_settings.preferences` or local state) is the build's call ([AOD-17](https://linear.app/thexap/issue/AOD-17) §10); this design owns the screens the gate routes to, not the gate logic.

- **The RevenueCat wiring is the build's.** This places the body and the CTA; the `Purchases.purchasePackage` call, the offering fetch, and the trial eligibility are the billing `type:tech-task`'s ([AOD-12](https://linear.app/thexap/issue/AOD-12) §5, section 11).
